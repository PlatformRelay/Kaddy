# Evidence — E14-S03 Nix VM deploy: mechanism proven, boot-to-serve NOT yet (ADR-0303) — 2026-07-19

**Story:** E14-S03 · **Change:** `openspec/changes/e14-nix-golden-images/` · **Status:** PARTIAL (honest)

## What IS proven

The **deploy mechanism** works for the Nix Marketplace app exactly as for Caddy (E13-S05): a
`gridscale_server` with a `gridscale_storage` created from `template.template_uuid =
3aa9777e-…` (the `kaddy-nix` consumer import) + a public IPv4 + the gridscale Public Network
**provisions and powers on** (`power=true`, `status=active`, Public Network + IP `185.241.34.52`
attached — byte-identical wiring to the Caddy VM that serves in ~20 s).

## What is NOT yet proven — the from-scratch boot contract (ADR-0303's flagged risk)

The Nix VM **does not serve** over `:80`/`:2019` within a 6-minute window — across **two** live
attempts, each provisioned + powered on cleanly, each torn down clean (0 orphans):

| Attempt | Image | Result |
| --- | --- | --- |
| v1 | `raw` (nixos-generators), no qemu-guest profile | `GET /` = 000 for 6 min; VM active |
| v2 | `raw` + **`profiles/qemu-guest.nix`** (virtio_blk/virtio_net in initrd) | `GET /` = 000 for 6 min; VM active |

The Caddy (Packer/Ubuntu) image serves in ~20 s over the *same* networking, so the edge/routing is
proven — the gap is **inside the from-scratch NixOS image booting on gridscale**.

## Diagnosis (ranked) — for the dedicated follow-up

Console access on gridscale is VNC (graphical), not API-readable, so the boot stage (GRUB / initrd /
DHCP / caddy) could not be observed live. Ruled out and ranked:

1. **Firmware mismatch — RULED OUT.** gridscale's default `hardware_profile` maps to **i440fx/BIOS**
   (`hardware_profile_config.machinetype ∈ {i440fx, q35_bios, q35_uefi}`; the GSK nodes run `default`).
   The image is BIOS/MBR (`raw` = MBR + GRUB), which **matches** — so `raw-efi` is *not* the fix.
2. **virtio missing — ADDRESSED, insufficient alone.** v2 added the qemu-guest profile; still no serve.
3. **`.gz` format mismatch — LEADING hypothesis.** The Caddy `.gz` was produced by
   `gridscale_snapshot.object_storage_export` (gridscale's native snapshot format); the Nix `.gz` is a
   plain `gzip` of the nixos-generators raw disk. If gridscale's Marketplace/storage import expects its
   own snapshot layout, the imported storage would be non-bootable even though it provisions "active".
   **Fix path:** build the Nix image *through* the gridscale snapshot-export path — boot a throwaway
   gridscale VM, `dd` the Nix raw image onto its disk (or `nixos-install` into it), snapshot →
   `object_storage_export` → register — so the `.gz` is byte-compatible with the Caddy one.
4. **Root-device detection.** The `raw` format labels root and sets `root=` on the kernel cmdline; a
   label/UUID mismatch vs gridscale's presented disk would stall initrd. Verify with console.
